# 読解デッキ｜アプリ作成構想

作成: 2026-07-13 / 状態: 動作試作品あり

## 1. 仮称

- アプリ・カード全体: **読解デッキ**
- 1つの長方形の項目: **読解カード**
- カードへ登録する知識: **読解項目**

「単語帳」では、単語以外の熟語、ディスコースマーカー、基本構文、数量表現を含めにくいため、この仮称を使います。名称は開発前に変更して構いません。

## 2. 目的

実際の英語長文で分からなかった知識を教材ごとに散らさず、1か所へ集めて復習できるようにします。

中心はテストの正誤判定ではなく、長方形のカードをパラパラめくり、本人の言葉に近い一文の重要部分だけを思い出してからタップで確認することです。問題文と回答欄へ分けず、短い標語へ圧縮しすぎない私的な一文を優先します。正確な説明と診断根拠は、別の詳細欄や実読報告へ残します。

## 3. 情報の役割分担

- `review_items.json`: 読解カードの元データ。教材横断の正本
- `実読チャット報告_YYYY-MM-DD.md`: 初読時の反応、白紙化、早期確定、保持失敗などの観測記録
- `weakness_data.md`: 複数セッション後に司令塔が統合した原因分析
- `読解カード_選定・まとめ方.md`: カードへ何を入れ、どう短くするかの基準
- このMD: アプリの画面・動作・開発順

GitHubの非公開リポジトリを情報の正本とします。実践チャットは確認した知識をGitHubへ書き込み、アプリはそのデータを読みます。アプリ内だけにしか存在しないカードを作らない方針です。

ユーザーが実践チャットで新しい画面案・動作・優先順位を直接伝えた場合は、そのチャットからこのMDを更新できます。

## 4. 中心画面

スマートフォンを優先し、画面中央へ長方形のカードを1枚表示します。

### デザイン方針

- iOS・Apple製品に近い、余白が広く、情報の階層が明確な外観にする
- 半透明、柔らかい影、控えめな境界線を使い、装飾よりカード内容を目立たせる
- 端末のライト／ダーク設定へ自然に追従できる構成にする
- カードは、標準的なトレーディングカードを横向きにした比率を基準にする。暫定値は約 `1.40:1`（横:縦）とする
- 画面幅に応じて縮小しても比率を保ち、片手操作できる余白を残す

```text
┌─────────────────────┐
│ Discourse marker   The Rules 1 L1 │
│                                     │
│             After all               │
│                                     │
│    ［ここをタップして重要部分を表示］   │
│                                     │
│             理由を示す               │  ← タップ後
│                                     │
│       詳細を見る / 元の文を見る        │
└─────────────────────┘
```

### 常に見えるもの

- 本人の言葉に近い自然な一文。ただし重要部分だけは下線の空欄
- 種類。例: ディスコースマーカー
- 出典。例: The Rules 1 Lesson 1

### 最初は隠すもの

- 一文の中で最も思い出したい部分
- 例: `After allは、＿＿＿＿を出すときに使う。` の `前の内容の理由`

下線部分をタップすると、周囲と同じ文字サイズで内容を表示します。再度タップして隠せるようにします。必ず文字入力させたり、厳密な日本語訳と照合したりはしません。

### 必要な時だけ開く詳細

- 正確な説明
- 最初にどう誤ったか
- 見分ける条件
- 元の英文断片
- 出典と作成日

カード表面を説明で埋めず、詳細は折りたたみます。

## 5. 基本操作

- タップ: 重要部分を表示・非表示
- 左右スワイプ、または前へ・次へボタン: カードをめくる
- スワイプ中は指に追従してカードが移動・傾き、離した速度に応じて加速・減速する
- 指を離した後は、iOS風のばね感を持って次の位置または元の位置へ吸着する
- めくる時は、わずかな回転、奥行き、影、背後の次カードの動きを組み合わせる
- 動きが苦手な場合に備え、端末の「視差効果を減らす」設定へ対応する
- 「もう一度」「思い出せた」: 復習結果を残す候補。MVPで採用するかは試作後に決める
- 星印: 特に重要なカードを残す
- 検索・絞り込み: 教材、Lesson、種類、作成日、学習状態

採点が負担になる場合は、最初の版では評価ボタンを付けず、タップとカード送りだけにします。

## 6. 画面候補

### MVPで必要

1. 今日見るカード、または全カードを始める入口
2. 1枚表示のカード画面
3. 一覧・検索画面
4. 教材・種類による絞り込み
5. 日付順、出典別、ランダムから復習方法を選ぶ入口

### 後から追加

- 復習間隔と「今日の復習」
- 星付き、苦手カードだけのデッキ
- カード追加・修正画面
- 白紙化、早期確定など原因分析との関連表示
- 教材別・種類別の蓄積状況

原因分析は重要ですが、最初のカード画面へ混ぜすぎません。知識の復習と原因分析はデータ上も画面上も別レイヤーにします。

## 7. データの流れ

```text
実際に長文を読む
  ↓
本人が段落を見返し、分からなかった箇所を申告
  ↓
実践チャットが正しい意味・機能を確認
  ├─ review_items.json：読解カード
  └─ 実読報告MD：読めなかった原因の証拠
  ↓
読解デッキがreview_items.jsonを読み込む
  ↓
タップして思い出し、カードをめくる
```

チャットからの入力はすべてGitHubへ残します。アプリを作り直しても、カード本体が失われない構成にします。

### 出典別の自動整理

AIがカードを登録する時に出典情報を構造化して保存し、アプリ側が自動的に棚・デッキへ振り分けます。ユーザーがカードを手作業で移動する運用にはしません。

- 参考書: 著者・シリーズ／書名・巻／Lesson・長文番号
  - 例: 関正生 → The Rules 1 → Lesson 1〜12
- 過去問: 大学／年度／学部・日程／大問・長文番号
  - 例: ○○大学 → 2024年度 → 一般入試 → 大問3
- 共通: 登録日、カード種別、作成元、学習状態

復習時は、次の入口を用意します。

- 日付順: 新しい順・古い順
- 出典別: 参考書や大学過去問のまとまりから選ぶ
- ランダム: 選択範囲内のカードを混ぜる

### カードのまとめ方を切り替える

カードを固定フォルダへ複製して保存するのではなく、同じカードデータを条件に応じて組み替えて表示します。

- 参考書別: `The Rules 1`だけをLesson順にめくる
- 学習日別: `2026-07-13`に実際に出会ったカードを、参考書をまたいでめくる
- 期間別: `2026-07-13〜07-19`など、1週間に出会ったカードをまとめてめくる
- 出典内ランダム: 参考書や大学過去問を選んだうえで順序だけ混ぜる
- 全体ランダム: すべてのカードから混ぜる

日付は、AIがカードをGitHubへ保存した日ではなく、ユーザーがその表現に実際に出会った学習日を基準にします。同じ知識へ別の日・別の教材で再度出会った場合に備え、カード本体とは別に「出会った日と出典」の履歴を持たせます。これにより、カードを重複作成せず、複数の日付デッキ・参考書デッキへ同じカードを出せます。

自動整理は「Webアプリだから自動で理解できる」のではなく、AIが登録時に出典項目を正しく付け、アプリがその項目を読んで分類することで成立します。

## 8. 技術方針

第一候補は、スマートフォンで使えるモバイル優先のWebアプリまたはPWAです。インストールに近い形でホーム画面から開ける構成を目指します。

ただし、非公開GitHubのデータを読むための認証情報をブラウザへ直接埋め込んではいけません。試作前に、次のどちらかを決めます。

- ローカルまたはビルド時にJSONを取り込む、読み取り中心の簡易版
- サーバー側でGitHubへ接続し、本人だけが使える同期版

最初からGitHubへの自動書き戻しまで作ると認証と競合処理が増えるため、MVPは読み取り中心にします。カード追加は当面、実践チャットからGitHubへ行います。

## 9. MVPの完成条件

1. `review_items.json` を読み込める
2. 1枚の長方形カードを表示できる
3. 重要部分をタップで表示・非表示にできる
4. 前後のカードへ移動できる
5. 教材・種類で絞り込める
6. スマートフォン幅で無理なく使える
7. `After all → 理由を示す` など短い私的メモが主表示になる
8. 横向きトレーディングカード比率と、慣性・ばね感のあるスワイプを試せる
9. 日付順、出典別、ランダムを切り替えられる

## 10. 開発順

1. Lesson 1〜3で、読解カードの選定・短縮基準を検証する
2. `review_items.json` を、画面用の短い表示と正確な詳細を分けられる構造へ更新する
3. 長方形カード1枚だけの試作品を作る
4. 実データを入れて、情報量とタップ位置を調整する
5. 一覧・検索・絞り込みを追加する
6. 必要なら復習間隔・履歴を追加する
7. GitHubへの自動同期方法を決める

## 11. 未決定事項

- 「読解デッキ」という名称を正式採用するか
- 英文断片を常時表示するか、詳細に隠すか
- 「もう一度／思い出せた」から次回表示までの間隔をどう決めるか
- 復習履歴・計画・分析を共通データベースへ移す時期と、GitHubバックアップの頻度
- オンラインで本人だけが使える認証方式
- 横向きカードの最終的な縦横比と、端末別の最大サイズ
- 日付・期間を選ぶ画面をカレンダー型、一覧型、または両方にするか

未決定事項は、実際のカードを見ながら決めます。MVPへ不要な機能は先に固定しません。

## 12. 最初の動作試作品

2026-07-13に `app/` へ実装しました。

- 起動直後からカードのプレビューを表示する
- 「前回の続き」を最も大きな入口として上部へ置く
- 参考書・日付・すべてでプレビューをその場で組み替える
- シャッフルと星付き絞り込みを使える
- 復習カードは約 `1.48:1` を基準にし、iPadでは少し横長、狭いiPhoneでは内容が切れないことを優先して自動調整する
- カードを指に追従させ、離した速度に応じて送り時間を変え、次のカードをばね感のある動きで入れる
- 答えはタップまで隠し、最初の誤り・手がかり・元の英文は詳細へ分ける
- 星と前回位置は当面その端末内へ保存する
- アプリ用データは正本のスナップショットであり、GitHubからの自動同期は未実装
- 実機フィードバックにより、端末のダークモードに追従せず白基調で固定した
- 問題文と回答欄を廃止し、自然な一文の中心だけを下線で隠す方式へ変更した
- 不要カードは即時・永久削除せず、端末内のゴミ箱へ移す。直後の取り消しとゴミ箱からの復元を備える
- プレビューからグラデーションと装飾円を外し、白い紙面、分類ラベル、通し番号、控えめな段差影で単調さを抑える
- マウス／トラックパッドではホバー、キーボードではフォーカス、タッチ端末では押下中に青い選択枠を表示する。タッチは二度押しにしない
- 星は復習カード右上の独立ボタンに残す。個別操作は縦三点メニューへまとめ、詳細表示・編集・ゴミ箱操作を配置する。ゴミ箱一覧への入口も独立して残す
- 空欄は答えの実寸を初めから確保し、表示前後で改行・文位置・カード位置を動かさない。「タップ」と答えだけを同じ領域で短く切り替える
- 編集画面ではカード文章を直接変更し、文章内で選択した範囲を隠す部分に設定できる。見出し・種類・詳細・出典も編集対象とする
- 新規カード追加は編集画面と共通化する。試作品では追加・編集を端末内へ保存し、正本同期・複数端末共有は後続段階で実装する

## 13. 採用した拡張機能

ユーザーが2026-07-13に採用した5機能を、端末内で試せる統合試作品へ実装した。

1. **今日の復習**: 答え表示後に「もう一度／思い出せた」を記録し、当日の復習対象を自動提示する
2. **AIカード確認待ち箱**: 実践チャットやAIが作ったカード候補を、採用・編集・不要の確認後に本番デッキへ入れる
3. **弱点分析画面**: A〜Gの欠落クラス、白紙化、早期確定、未自動化などを、事例数と確度つきで表示する
4. **自動バックアップ**: 現版は端末内のJSONへ自動保存し、手動で書き出せる。GitHubへの自動保存はオンライン版で接続する
5. **同期状態表示**: 現版は「この端末・保存中／保存済み」を表示する。オフライン、失敗、他端末更新はオンライン同期と同時に追加する

## 14. 読解デッキから英語学習司令画面へ

読解カードは、長文を読めるようにする方法の一部である。元の計画には、初見読解、精読、A〜Gの欠落分析、白紙化・早期確定等の処理分析、参考書への戻り、音読・再読、月間→週間→日別計画が含まれる。したがって、カード画面へ全情報を詰め込まず、1つのアプリの中で次の領域を分ける。

- **今日**: 今日の参考書、Lesson、復習、再読、時間不足時の縮小案
- **実読**: 実践チャットへの入口、現在の長文、セッション報告、確認待ちカード
- **復習**: 現在の読解デッキ。カード、今日の復習、星、出典・日付別デッキ
- **分析**: A〜Gの欠落クラス、白紙化、早期確定、未自動化、根拠事例、推移
- **計画**: 志望校と参考書ルート、月間目標、週間計画、日別カレンダー、再判定日

アプリは記録・表示・復習・日程管理を担当する。長文中の追加質問、診断の再解釈、参考書ルートの大きな変更はAIが行い、アプリへは提案として送り、ユーザーまたは司令塔が承認して反映する。長い会話や根拠全文はGitHubのMDへ残し、アプリは要点と次の行動を表示する。

### 計画の扱い

- `plan.md` の月間方針、`week1.md` の週間・日別計画、`route.md` の参考書役割を共通データへ構造化する
- 計画変更はAIが候補を作るが、自動確定しない
- 未実施項目は単純に失敗扱いせず、次の日へ移動する、量を減らす、再判定待ちにする、の選択肢を出す
- 15〜20分、60〜90分、約2時間、通常日の縮小規則をカレンダーから選べるようにする

### 実装順

1. 共通データベース、同期状態、編集履歴、自動バックアップ
2. 今日の復習と復習履歴
3. AIカード確認待ち箱
4. 実読報告から弱点事例を取り込む分析画面
5. 参考書ルート・月間・週間・日別カレンダー
6. AIが計画変更案を作り、ユーザーが承認する流れ

### 2026-07-13 データ基盤・長文再読

- 今日・週間計画は端末の日付から選び、固定の7月13日表示をやめる。未完了は失敗にせず、元の計画日を保持したまま翌日へ出す。
- 使える時間は「15〜20分・60〜90分・約2時間・通常」の4段階とし、短時間日は新規初見を急いで消化しない。
- 復習は「読解カード・長文再読・処理練習」の3系統に分ける。
- 長文再読は1長文につきまず30回。基本は1日5回ずつ6日に分け、各回の実施日と停止の有無を保存する。所要時間は保存しない。
- 1〜10回は文法・構文、11〜20回は意味・内容、21〜30回は理解できる範囲での速度を意識する。アプリは現在回数から「次の5回で意識すること」を表示する。
- 必ず声に出し、意味の切れ目で息継ぎする。`//` は必ず切り、`/` は切ってもよいという最低限の区切りルールも表示する。
- 30回で追加記録を止めて卒業判定を行う。必要になった場合だけ卒業後に再開する。
- 30回読みは、英語を英語のまま処理する自動化を狙う正式な学習工程として扱う。回数だけで効果を断定せず、停止の有無と、構造・内容・速度の処理変化で検証する。
- 初読済み長文の共有正本はプロジェクト直下の `study_seed.json` の `rereadItems` とする。Codex・実践チャット等のAIは、長文名、参考書、Lesson、初読日、出典報告をここへ追記できる。
- アプリは共有正本へ新しく追加された長文と端末内データをIDで統合し、端末内に蓄積した音読回数・停止記録・卒業状態を消さない。
- カード復習は最新状態に加えて全履歴を蓄積する。カードの日付分類はデータ作成日ではなく実際の学習日を使う。
- 端末内の共通学習データはJSONで書き出し・復元できる。オンライン同期と競合処理は公開段階で追加する。

### 実読チャットとのデータ接続

- 教材横断カードの共有正本はプロジェクト直下の `review_items.json` とする。
- 読了長文・計画・弱点表示の共有正本はプロジェクト直下の `study_seed.json` とする。
- `app/public/review_items.json` と `app/public/study_seed.json` はアプリ配信用コピーであり、実践チャットは直接編集しない。
- アプリの `dev`、`build`、`test` の前に同期スクリプトを実行し、2つの正本を検証して `app/public` へコピーする。必須項目が欠けたカードは、そのまま表示して壊すのではなく同期時にエラーにする。
- 実践チャットは進行中の `実読チャット報告_YYYY-MM-DD.md`、`review_items.json`、`study_seed.json` の3つだけを更新すればよい。アプリ本体のソースコードを共有する必要はない。
- `study_seed.json` では、読了した全Lessonを `completedLessons` に置き、30回読みへ選定した長文だけを `rereadItems` に置く。読了と30回読み選定を同一扱いにしない。
- 実読チャットは既存 `study_seed.json` を別スキーマで作り直さず、計画・弱点・既存の再読対象を保持して追記する。
- GitHub上の変更をローカルアプリへ取り込む処理は、現段階では自動常時同期ではない。GitHubの最新版をローカル正本へ反映したうえでアプリを再起動・再ビルドする。オンライン自動同期は公開段階の別機能として扱う。
